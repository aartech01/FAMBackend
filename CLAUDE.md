## Architecture

This is an **Express 5 + MongoDB + Redis** backend using ES modules (`"type": "module"` in package.json). The entry point is `server.js`.

### Three-role auth model

There are three separate user types, each with their own Mongoose model:
- **Admin** — platform administrators, password-based login, JWT issued on `/api/admin/login`
- **Organizer** — event organizers created by admins, password-based login, JWT on `/api/organizer/login`
- **User** — end users, **OTP-only login** (no password), JWT issued on `/api/auth/verify-otp`

`middleware/auth.js:verifyToken` dispatches to the right model based on the `role` claim inside the JWT. There are two auth middlewares: `middleware/authMiddleware.js` (JWT-only, no DB lookup) and `middleware/auth.js` (full DB lookup with block/active checks) — prefer `auth.js` for protected routes. `verifyTokenSimple` in `auth.js` is a synchronous helper used by Socket.IO (returns decoded payload or `null`).

### Request flow

```
server.js
  → helmet + cors + express.json + apiRateLimiter (global)
  → routes/*Routes.js
    → middleware/auth.js (verifyToken) + middleware/roleMiddleware.js (isAdmin/isOrganizer)
    → controllers/*Controller.js
      → services/ (business logic)
      → models/ (Mongoose)
```

### Key domain concepts

**Events** (`models/Event.js`) — created by admins; each creation automatically creates an Organizer record with a temp password (emailed to `organizerEmail`). Users join via a unique `eventCode` or by scanning a QR code generated by `services/qrService.js`. `approvalMode` controls join flow:
- `"auto"` — users join immediately as participants
- `"manual"` — users land in `pendingApprovals`; organizer must call `PATCH /api/organizer/approve/:userId`

**Family Tree** — per-event relationship graph. `models/Relationship.js` stores directed edges between `User` pairs with a `familySide` field (`"groom"`, `"bride"`, or `"common"`). `services/treeEngine.js` (a static class) handles:
- Validation: duplicate check, circular-relation detection (BFS ancestor traversal), parent/child age-gap enforcement
- Tree generation: produces nodes/edges and Mermaid flowchart code, saves snapshots to `models/TreeHistory.js`
- Each user can add at most 4 people (`User.canAddPerson()` method, `addedPeopleCount` field)
- `generateFullTree` splits into groom/bride sub-trees for `treeType === "wedding"` or `"anniversary"`, otherwise returns a single `"common"` tree

> **Known schema gap**: `treeEngine.js` references `event.treeType` and `event.treeConfig` (holding `mainPersonId`, `groomId`, `brideId`), but these fields are absent from the `Event` schema. Mongoose `strict: true` silently drops them on save, so this part of the tree logic is currently inoperative and needs schema additions before use.

**Organizer model** (`models/Organizer.js`) — has `assignedEvent` (single ObjectId), `validTill` date, `permissions[]` enum, and `accessCode`. A daily midnight cron deactivates organizers whose `validTill` has passed and marks their event inactive.

**User.isTemporary** — set `true` when a user record is created at OTP-send time (before profile completion). `username` and `dob` are not required on temporary users. The OTP and expiry are stored on the User model in `otp`/`otpExpiry` fields (both `select: false`).

**WorldTree** (`models/WorldTree.js`) — a premium, payment-gated personal tree separate from event trees. Has a 1-year `validTill` validity and `privacyMode` ("public"/"private"/"family-only"). `services/paymentService.js` is an **unimplemented stub** — payment gateway integration (Razorpay/Stripe) has not been built yet.

**Notifications** (`services/notificationService.js`) — hybrid: persisted to `models/Notification.js`, delivered real-time via Socket.IO (`ioInstance.to('user_<id>').emit(...)`), and optionally sent by email. The `io` instance is injected via `setSocketIO(io)` called in `server.js` before routes load. Cron jobs in `cron/jobs.js` trigger birthday/anniversary/event-reminder notifications daily.

**Cloudinary** (`config/cloudinary.js`) — used for user profile image uploads and for storing tree Mermaid code as raw text blobs. Configured via three `CLOUDINARY_*` env vars.

**Audit logging** (`services/auditService.js` → `models/Log.js`) — polymorphic `refPath` on `userModel` field so the `userId` reference resolves to Admin, Organizer, or User. Logging failures are swallowed (never throw) to avoid breaking main flows. Logs older than 30 days are deleted by a monthly cron job.

```

```

### Rate limiting

Four tiers in `middleware/rateLimiter.js`:
- `apiRateLimiter` — 100 req/min, applied globally in `server.js`
- `loginRateLimiter` — 5 attempts/15 min, applied to OTP send/verify and admin/organizer login routes
- `strictRateLimiter` — 10 req/hour, for sensitive operations
- `roleBasedRateLimiter(role)` — factory for per-role limits

### Security notes

`middleware/security.js` does **not** use `express-mongo-sanitize` or `xss-clean` — both packages attempt to overwrite `req.query`, which is a read-only getter in Express 5 and throws at startup. XSS/injection prevention relies on Helmet headers, Mongoose parameterized queries, and the inline `sanitizeInput` helper exported from that file.

### Patterns to follow

- All controllers return `{ success: true|false, message, data? }` shaped JSON.
- Old code is left commented out in many files (do not delete or uncomment unless intentional — the live code follows the commented block).
- Seed scripts (`adminSeed.js`, `organiserSeed.js`) connect/disconnect MongoDB themselves and are run standalone with `node`.
- Socket.IO authentication uses the JWT from `socket.handshake.auth.token`, same secret as HTTP.
- **BullMQ** is listed as a dependency but is currently unused — do not wire it up unless intentionally adding a job queue.
